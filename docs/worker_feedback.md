# Agent Feedback

> Agent → 用户的异步消息通道。Worker / Auditor / 其他 agent 均可写入。
> 三种类型：
> - `[决策]`：需要用户判断的设计问题，阻塞对应 backlog item（转 `waiting-feedback`）
> - `[通知]`：值得用户了解的信息，不阻塞工作（实现取舍、跳过步骤的原因、潜在改进点等）
> - `[观察]`：不算 bug 但值得注意的现象（代码异味、设计不一致、潜在改进方向等）
>
> Agent session 很长，用户无法回看全部过程。这里是 agent → 用户的异步摘要。
> Discussion agent 在每次对话开始时呈现，用户确认后删除。

- [决策] **#151 修复方向（三案利弊，待拍板）**：
  - **(a) per-use drop（D1 模式一致）**：合成站点后按消费完成发 drop。障碍：dict TUPLE 的 drop_table 是 no-op fallback（slots 不递归）→ 需新 runtime helper `ring_dict_drop(dict, n)` 或逐 slot 发 drop；合成在 codegen 层、HIR/ANF 永远不可达，配对须全在 codegen 内做（B-104b cond-box drop 模式可援引），4 条合成路径 × 各消费形态配对面大；**churn 不消**——每泛型派发 4 alloc + 4 free，热循环放大（自编译尺度 ~127M 对）。
  - **(b) 全局缓存单例（推荐）**：codegen 级——对模块中静态出现的每个 Simple dict name 合成一次、存 LLVM global（module-init eager 或首用 lazy-init 均可），使用点 = 单 load。**与 JS 后端模型精确对齐**（dict 语义上本就是 per-(Type,Trait) 静态实体）；TUPLE/CLOSURE/STR 三成员 + churn 一次全消。**与 R-clean「never-drop 不回收」教训不冲突**：Type intern 被否是因 Type 随推断步无界新生；dict 集 = 程序文本静态属性（#impl×#trait + builtin 组合），有界小常数 → 真等价于安全 intern。**安全前提已核**：lambda 捕获路径 dup/drop 平衡（gen_lambda 构造时 `gen_dup_value`、`drop_closure_env` 释放 → 单例 rc 永 ≥1）；dictwrap/wrapped env count=0 不 drop。配套两件：B-103 分类表 `ring_get_builtin_dict` FRESH → 改注 BORROW-returning（共享单例）；建议 dedicated never-drop typeid 作纵深（stray drop 变 no-op）。**不覆盖的残留**：wrapped dict inner 可为动态 dict 参数 → 仅静态 Simple 可缓存，wrapped 维持 per-site（CLOSURE_ENV 量级小，日后可见再收）；Ord cmp 结果 INT box 是独立 codegen 缺口，需 while-cond box drop 同模式补一处。
  - **(c) dict 提升为 HIR 一等 evidence 值**：D1/D2 统一覆盖、verifier 豁免类消失，架构最彻底；但 checker/HIR/双后端管道全动，churn 与收益不成比 → 长期方向，不建议本轮。
  - **推荐 = (b) + Ord-result-box 补 drop**：一次合成消三类泄漏 + 消 churn + 对齐 JS oracle 模型，有界性论证成立；(a) 模式留作 wrapped-dict 残留的后续手段。
- [观察] **tid13 SB（StringBuilder）= 12.0M residual（rank 6 新面孔）**：字符串插值 builder 泄漏嫌疑——#151 之外下一个可见类，待 #151 收口后归因。
